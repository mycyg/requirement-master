# 代码审查报告与功能优化建议

审查日期：2026-05-26  
范围：`web/` 前端、`app/` FastAPI 后端、上传/交付/AI 自动处理核心链路。

## 2026-05-29 UI / 客户端 / 数据流复核

本轮从两条线复核：

- UI 交互：Web、客户端 Web 壳、真实 Tauri exe 窗口、移动端、桌面、超宽屏截图。
- 数据流：项目生命周期向需求、澄清、评论、活动、交付、日程、网盘、知识库、通知和客户端同步的下游传播。

已修复：

- 归档或软删除项目后，子需求的澄清对话、评论和活动入口会直接返回 404，避免项目被封存后仍能继续补充需求或写入协作流。
- 澄清流后台生成 summary 时再次确认父项目仍处于 active 状态，避免 LLM 长任务结束后把已归档项目里的需求重新推进状态。
- Tauri fresh config 不再把空 IP 计算成 `http://:8080`，避免后台 SSE / reminder 在首次配置前反复访问无效 URL。
- 客户端 Onboarding 移除“双向同步”可选项；当前 Rust 同步层只开放 `off / download`，UI 不再展示后端会拒绝的模式。
- Tauri 默认打包目标从 `msi + nsis + 跨平台包` 收敛为 Windows 可通过的 NSIS，`npm run tauri:build` 可直接产出安装包。

验证结果：

- `python -m compileall app client scripts asr_service tts_service` 通过。
- `python scripts/smoke_workflow.py` 通过，并新增“项目归档后封住子需求 chat/comments/activity/answer”的回归覆盖。
- `npm run build` 通过。
- `npm run e2e:web` 通过：25 个用例中 23 passed / 2 skipped；覆盖 Web 桌面、移动端和超宽屏。
- `cargo check --manifest-path client-tauri/src-tauri/Cargo.toml` 通过。
- `npm run tauri:build --workspace=client-tauri` 通过，输出 `client-tauri/src-tauri/target/release/bundle/nsis/需求管理大师_0.2.0_x64-setup.exe`。
- 客户端专项 E2E 通过：`client-routes.spec.ts` 与 `client-spaces.spec.ts` 共 2 passed，并刷新客户端截图。
- 真实 Tauri exe 已启动并截图：`screenshots/aurora/client/native-tauri-window-clean.png`。

数据流合理性结论：

- 主干层级“项目 -> 需求 -> 接单方 -> 工作区 -> 交付/验收”现在是闭环的；项目归档/删除会向下游读写入口传播。
- 派活端与接活端的职责分界基本合理：Web 保持派活/查看/验收，本地端承担接单、工作区编辑、同步和交付。
- 当前仍刻意不开放双向同步，数据安全优先；UI 已与实现保持一致。
- 知识库继续采用可 grep 的项目语料，不引入 embedding；这与当前内网小团队、可追溯证据优先的定位一致。

## 本轮已修复

- 摘要生成后不再直接进入 `ready`，改为 `summary_ready`，必须由提交人确认投递或选择 AI 处理。
- 持久化接单人 `claimed_by_user_id / claimed_by_nickname`，交付上传只允许当前接单人执行。
- 收紧通用状态接口，验收、返工、AI 处理和交付完成改走专用接口，避免绕过必要业务记录。
- 移除自动 AI 工具里的直接 `run_bash` 能力；当前只允许列文件、读写工作目录文件和提交结果，避免服务进程执行任意 shell。
- 生产环境会拒绝默认 `COOKIE_SECRET` 和通配 CORS，cookie 增加 `COOKIE_SECURE` 配置。
- AI 后台任务异常会自动回退到 `ready`，记录 `ai_failed` 并推送状态更新。
- 人工交付上传后先进入 `delivery_doc_pending`，交付文档生成完成后再进入 `delivered`，详情页会收到 SSE 刷新。
- 托盘客户端只展示“我接的单”作为交付候选，并对每个上传分片调用 `raise_for_status()`。
- 附件上传和交付上传增加 chunk 数校验、单片大小限制、重复 chunk 拒绝和 finalize 阶段的 chunk 完整性校验。
- 服务启动时会清理超过 24 小时的 `_partial` 上传残留目录。
- 交付 ZIP 增加路径、条目数、单文件大小、总解压大小和压缩比校验；ZIP 内单文件下载改为流式读取。
- 需求列表接口改为 join 一次取回项目 slug 和提交人昵称，避免看板列表 N+1 查询。
- 需求编号生成捕获唯一约束冲突并重试，降低并发提交时编号撞车的失败率。
- 澄清页刷新后可以从历史 summary 恢复确认投递卡片，避免停在普通聊天气泡。
- 提交、澄清、AI 自动处理、验收和返工增加提交人权限校验。
- 附件列表、附件下载和同步清单增加访问控制：草稿阶段仅提交人可见，接单后仅提交人和接单人可见。
- 草稿、澄清中、待确认投递的需求记录、澄清对话、评论和活动默认仅提交人可见；投递后才进入公开协作视图。
- 交付包列表、ZIP 包下载和 ZIP 内单文件下载增加提交人/接单人权限校验。
- 附件分片上传、交付分片上传都绑定发起用户，防止其他登录用户复用 `upload_id` 注入分片或 finalize。
- 同步 ACK 改为仅提交人或接单人可写，避免任意看板用户把全局同步状态误标为已同步。
- 澄清 SSE 增加同一需求并发锁，避免 React 严格模式或重复点击造成双 LLM 回合、重复问题和状态竞争。
- 澄清页可以从最新历史问题恢复可操作卡片；流式完成后隐藏草稿推理片段，避免向用户展示不稳定的模型中间文本。
- 澄清页、确认投递、接单和开始处理按钮增加忙碌态、错误态和防重复提交。
- 提示词进一步收紧 AI 可执行边界：提示词仍为英文，并明确要求使用用户语言输出；AI 不能执行 shell、安装依赖、联网或做运行时测试。
- AI 自动交付文档会根据需求语言输出中文或英文标题，避免英文需求收到中文交付说明。
- 新增 `scripts/smoke_workflow.py`，覆盖身份、项目、需求、附件权限、投递、同步、接单、开始处理、取消和分片归属校验。

## 已完成的 UI/UX 优化

- 全局改为微软雅黑字体，并建立 Anthropic 风格的温暖纸面色、细网格背景、克制边框、8px 圆角和统一控件类。
- 引入 `lucide-react`，替换大部分 emoji/文本符号按钮，提升可识别性和一致性。
- 看板支持超宽屏 5 列布局，移动端单列堆叠；并将状态刷新请求改为并行。
- 澄清页从固定双栏改为移动端单栏、桌面双栏，避免 390px 宽度水平溢出。
- 需求详情页 tabs 增加横向滚动与图标，移动端不再挤压。
- 上传、语音、评论、交付、设置弹窗都补了移动端布局和错误态展示。
- 澄清页状态展示复用统一状态徽标，不再直接暴露 `doing / summary_ready` 等内部枚举。
- 浏览器 E2E 覆盖了创建项目、提交需求、AI 澄清、总结确认、人工投递、接单、开始处理、看板流转、移动端与 2560px 超宽屏无横向溢出。

## 功能优化建议

1. 增加「我的需求 / 我接的单 / 待我处理」视图，减少看板信息噪音。
2. 为需求增加 SLA、截止时间、负责人、标签和筛选排序，支持超时高亮。
3. 给 AI 澄清增加「问题进度」和「一键重新生成总结」，让用户更容易控制结果。
4. 上传附件后展示解析状态、失败原因和可重新解析按钮。
5. 交付验收增加差异摘要、文件预览和返工原因模板。
6. 加项目级权限、只读访客和管理员角色，避免内网环境里所有人都能看到所有需求。
7. 增加服务健康页，统一展示 LLM、ASR、TTS、SSE、数据库和磁盘容量。
8. 引入数据库迁移流程，避免 `Base.metadata.create_all` 在结构演进时失控。
9. 增加需求模板库和字段化验收标准，让常见工作台需求可以更快进入可派发状态。
10. 增加审计日志导出和敏感附件水印，方便管理层追踪跨团队交付过程。
11. 给 AI 澄清增加“采纳/忽略本问题”和“补充上下文后重写总结”，减少用户被单一路径卡住的概率。
